# Агенты
create_agent предоставляет готовую к использованию реализацию агента.

## Основные компоненты:

### Модель — это механизм логического вывода вашего агента. Она может быть задана несколькими способами, поддерживающими как статический, так и динамический выбор модели.

#### Статическая модель. Настраиваются один раз при создании агента и остаются неизменными на протяжении всего времени работы. Это самый распространенный и простой подход.
Полный список соответствий строк идентификаторов моделей "openai:gpt-5" приведен в справочнике.
Т.к. у меня уже есть готовая модель, я указываю "agent = create_agent(model, tools=tools)"

#### Динамическая модель. Выбираются во время выполнения в зависимости от текущего состояния и контекста. Это позволяет использовать сложную логику маршрутизации и оптимизировать затраты.
Чтобы использовать динамическую модель, создайте промежуточное ПО с помощью декоратора @wrap_model_call, который изменяет модель в запросе:
Предварительно связанные модели (модели с уже вызванным bind_tools методом) не поддерживаются при использовании структурированного вывода.

### Инструменты - дают агентам возможность совершать действия. Агенты не ограничиваются простым связыванием инструментов с моделью, а способствуют:
-- Последовательное выполнение нескольких вызовов инструмента (по одному запросу)
-- Параллельное выполнение вызовов инструмента при необходимости
-- Динамический выбор инструмента на основе предыдущих результатов
-- Логика повторных вызовов инструмента и обработка ошибок
-- Сохранение состояния при выполнении вызовов инструмента

#### Статические инструменты - определяются при создании агента и остаются неизменными на протяжении всего выполнения. Это самый распространенный и простой подход.
Чтобы создать агента со статическими инструментами, передайте ему список инструментов.
Инструменты можно задавать в виде обычных функций Python или корутин.
С помощью декоратора инструментов можно настраивать названия, описания, схемы аргументов и другие свойства инструментов.
Если указан пустой список инструментов, агент будет состоять из одного узла LLM без возможности вызова инструментов.

#### Динамические инструменты.
При использовании динамических инструментов набор инструментов, доступных агенту, изменяется во время выполнения, а не задается заранее. Не все инструменты подходят для любой ситуации. Слишком большое количество инструментов может перегрузить модель (контекст перегружен) и привести к ошибкам, а слишком малое — ограничить возможности. Динамический выбор инструментов позволяет адаптировать доступный набор инструментов в зависимости от состояния аутентификации, прав пользователя, флагов функций или этапа диалога.
Существует два подхода в зависимости от того, известны ли инструменты заранее:
1. Фильтрация предварительно зарегистрированных инструментов
    Если все возможные инструменты известны на этапе создания агента, их можно предварительно зарегистрировать и динамически фильтровать те из них, которые будут доступны модели, в зависимости от состояния, разрешений или контекста.
    1.1 Зависимость от состояния:
    Включайте расширенные инструменты только после достижения определенных этапов общения.
    1.2 Зависимость от разрешения:
    Фильтрация инструментов на основе пользовательских настроек или флагов функций в разрешениях.
    1.3 Зависимость от контекста:
    Фильтрация инструментов на основе прав доступа пользователей из контекста выполнения.
2. Регистрация инструмента во время выполнения
    Когда инструменты обнаруживаются или создаются во время выполнения (например, загружаются с сервера MCP, генерируются на основе пользовательских данных или извлекаются из удаленного реестра), необходимо зарегистрировать инструменты и динамически управлять их выполнением.
    Для этого требуются два промежуточных программных модуля:
        wrap_model_call - Добавьте динамические инструменты в запрос
        wrap_tool_call - Обеспечьте выполнение динамически добавленных инструментов
    Этот подход лучше всего использовать в следующих случаях:
        Инструменты обнаруживаются во время выполнения (например, на сервере MCP)
        Инструменты генерируются динамически на основе пользовательских данных или конфигурации
        Вы интегрируетесь с внешними реестрами инструментов
    Этот wrap_tool_call хук необходим для инструментов, зарегистрированных во время выполнения, поскольку агенту нужно знать, как запускать инструменты, которых не было в исходном списке. Без него агент не сможет вызвать динамически добавленный инструмент.

### Обработка ошибок инструмента
Чтобы настроить обработку ошибок инструмента, используйте @wrap_tool_call декоратор для создания промежуточного программного обеспечения
Агент вернет ToolMessage с пользовательским сообщением об ошибке в случае сбоя инструмента.

### Использование инструмента в цикле ReAct
Агенты следуют шаблону ReAct («Рассуждения + действия»), чередуя короткие этапы рассуждений с целевыми вызовами инструментов и используя полученные результаты для принятия последующих решений, пока не получат окончательный ответ.

### System prompt
Вы можете повлиять на подход вашего агента к выполнению задач, предоставив ему подсказку. Параметр system_prompt можно указать в виде строки.
Если system_prompt не указан, агент будет определять задачу непосредственно по сообщениям.
Параметр system_prompt принимает значение str или SystemMessage. Использование SystemMessage дает больше возможностей для управления структурой подсказок, что полезно для функций, специфичных для конкретного провайдера, таких как кэширование подсказок в Anthropic.
Поле `cache_control` с `{"type": "ephemeral"}` указывает Anthropic на необходимость кэширования этого блока контента, что снижает задержку и затраты при повторных запросах, использующих одну и ту же системную команду.

### Dynamic system prompt
Для более сложных случаев, когда необходимо изменить системную подсказку в зависимости от контекста выполнения или состояния агента, можно использовать промежуточное ПО.
Декоратор `@dynamic_prompt` создает промежуточное ПО, которое генерирует системные подсказки на основе запроса к модели.

### Name
Задайте необязательное `name` для агента. Это используется в качестве идентификатора узла при добавлении агента в качестве подграфа в многоагентные системы.
Предпочтительно использовать `snake_case` для названий агентов (например, `research_assistant` вместо `Research Assistant`). Некоторые поставщики моделей отклоняют названия, содержащие пробелы или специальные символы, с выдачей ошибок. Использование буквенно-цифровых символов, знаков подчеркивания и дефисов обеспечивает совместимость со всеми поставщиками. То же самое относится к названиям инструментов.

## Invocation (Вызов)
Вы можете вызвать агента, передав ему обновление `State`. Все агенты содержат последовательность сообщений в своем состоянии; чтобы вызвать агента, передайте ему новое сообщение.
Инструкции по потоковой передаче шагов и/или токенов от агента см. в руководстве по потоковой передаче.
В остальном агент использует Graph API LangGraph и поддерживает все связанные с ним методы, такие как `stream` и `invoke`.

## Передовые концепции

### Структурированный вывод
В некоторых ситуациях вам может понадобиться, чтобы агент возвращал результат в определенном формате. LangChain предоставляет стратегии для структурированного вывода с помощью параметра `response_format`.

`ToolStrategy` (from langchain.agents.structured_output import ToolStrategy) использует искусственный вызов инструмента для создания структурированного вывода. Это работает с любой моделью, поддерживающей вызов инструмента. `ToolStrategy` следует использовать, когда собственный структурированный вывод провайдера (через `ProviderStrategy`) недоступен или ненадежен.


`ProviderStrategy` (from langchain.agents.structured_output import ProviderStrategy) использует встроенную в провайдер модели генерацию структурированного вывода. Это более надежный подход, но он работает только с провайдерами, поддерживающими встроенную генерацию структурированного вывода.
Начиная с версии `langchain 1.0`, при простой передаче схемы (например, `response_format=ContactInfo`) по умолчанию используется `ProviderStrategy`, если модель поддерживает собственный структурированный вывод. В противном случае используется `ToolStrategy`.

### Память
Агенты автоматически сохраняют историю диалогов в состоянии сообщений. Вы также можете настроить агента на использование пользовательской схемы состояния для сохранения дополнительной информации во время диалога.
Информацию, хранящуюся в состоянии, можно рассматривать как кратковременную память агента: Пользовательские схемы состояния должны быть `AgentState` расширены `TypedDict`.
Существует два способа определения пользовательского состояния:
1. Через промежуточное ПО **(предпочтительно)**
2. Через `state_schema` на `create_agent`

#### Определение состояния с помощью промежуточного программного обеспечения
Используйте промежуточное программное обеспечение для определения пользовательского состояния, если к нему должны обращаться определенные хуки и инструменты промежуточного программного обеспечения.

#### Определение состояния с помощью `state_schema`
Используйте `state_schema` параметра в качестве ярлыка для определения пользовательского состояния, которое используется только в инструментах.

**Начиная с langchain 1.0, пользовательские схемы состояний должны быть TypedDictтипами. Модели и классы данных Pydantic больше не поддерживаются. Подробнее см. в руководстве по миграции на версию 1.**
**Определение пользовательского состояния с помощью промежуточного программного обеспечения предпочтительнее, чем с помощью state_schema на create_agent , поскольку это позволяет концептуально ограничить расширения состояния соответствующими промежуточными программами и инструментами.**
**state_schema по-прежнему поддерживается для обеспечения обратной совместимости на create_agent.**

### Потоковая передача
Мы увидели, что агента можно вызвать с помощью `invoke` для получения окончательного ответа. Если агент выполняет несколько действий, это может занять некоторое время. Чтобы отслеживать промежуточные результаты, мы можем выводить сообщения по мере их поступления.

### Промежуточное ПО
Промежуточное ПО обеспечивает широкие возможности для настройки поведения агента на разных этапах выполнения. Промежуточное ПО можно использовать для:
- Обработка состояния перед вызовом модели (например, обрезка сообщения, внедрение контекста)
- Изменение или проверка ответа модели (например, ограничения, фильтрация контента)
- Обработка ошибок при выполнении инструмента с помощью пользовательской логики
- Реализация динамического выбора модели на основе состояния или контекста
- Добавление пользовательских функций ведения журнала, мониторинга или аналитики
Промежуточное программное обеспечение легко интегрируется в работу агента, позволяя перехватывать и изменять поток данных в ключевых точках без изменения основной логики агента.
