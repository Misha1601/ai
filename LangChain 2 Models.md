# Модели

LLM — это мощные инструменты искусственного интеллекта, которые могут интерпретировать и генерировать текст, как это делают люди. Они достаточно универсальны, чтобы писать тексты, переводить с одного языка на другой, делать краткие изложения и отвечать на вопросы, не требуя специализированного обучения для выполнения каждой задачи.
Помимо генерации текста, многие модели поддерживают:
- Вызов инструмента — вызов внешних инструментов (например, запросов к базам данных или API) и использование результатов их работы.
- Структурированный вывод — когда ответ модели должен соответствовать определенному формату.
- Мультимодальность — обработка и возврат данных, отличных от текста, таких как изображения, аудио и видео.
- Рассуждения — модели выполняют многоэтапные рассуждения, чтобы прийти к определенному выводу.

Модели — это механизм логического вывода агентов. Они управляют процессом принятия решений агентом, определяя, какие инструменты вызывать, как интерпретировать результаты и когда выдавать окончательный ответ.

Качество и возможности выбранной вами модели напрямую влияют на базовую надежность и производительность вашего агента. Разные модели лучше справляются с разными задачами: одни лучше следуют сложным инструкциям, другие — структурированным рассуждениям, а некоторые поддерживают более широкие контекстные окна для обработки большего объема информации.

Стандартные интерфейсы моделей `LangChain` предоставляют доступ к множеству различных интеграций с провайдерами, что позволяет легко экспериментировать и переключаться между моделями, чтобы найти наиболее подходящую для вашего сценария использования.

## Основное использование
Модели можно использовать двумя способами:
- С помощью агентов — модели можно динамически настраивать при создании агента.
- Автономно — модели можно вызывать напрямую (вне цикла работы агента) для решения таких задач, как генерация текста, классификация или извлечение данных, без использования фреймворка для работы с агентами.

Один и тот же интерфейс модели работает в обоих контекстах, что дает возможность начинать с простого и при необходимости переходить к более сложным рабочим процессам на основе агентов.
​
### Инициализируйте модель
Самый простой способ начать работу с отдельной моделью в LangChain — использовать init_chat_model для инициализации модели из выбранного вами поставщика чат-моделей (примеры ниже):
`pip install langchain-openai`
Так как у меня уже есть готовая модель я буду использовать её, выполнив:
```
from models import MODEL_Y1
model = MODEL_Y1
```

### Поддерживаемые провайдеры и модели
`LangChain` поддерживает всех основных провайдеров моделей с помощью специальных пакетов интеграции. Каждый пакет провайдера реализует один и тот же стандартный интерфейс, поэтому вы можете менять провайдеров без необходимости переписывать логику приложения. Новые имена моделей работают сразу — обновление LangChain не требуется, — поскольку пакеты провайдеров передают имена моделей непосредственно в API провайдера.
Ознакомьтесь с полным списком поддерживаемых провайдеров или с разделом Провайдеры и модели, чтобы получить общее представление о том, как провайдеры, пакеты и имена моделей взаимодействуют в LangChain.

### Ключевые методы
Invoke, Stream, Batch
Помимо чат-моделей, `LangChain` поддерживает другие смежные технологии, такие как встраиваемые модели и векторные хранилища. Подробнее см. на странице с информацией об интеграции.

## Параметры
Чат-модель принимает параметры, которые можно использовать для настройки ее поведения. Полный набор поддерживаемых параметров зависит от модели и поставщика, но к стандартным относятся:
​
`model` `string` `required`
Название или идентификатор конкретной модели, которую вы хотите использовать с провайдером. Вы также можете указать модель и ее провайдера в одном аргументе в формате `:`, например `openai:o1`.
​
`api_key` `string`
Ключ, необходимый для аутентификации у поставщика модели. Обычно он выдается при регистрации для доступа к модели. Часто доступ к нему осуществляется путем установки переменной среды.
​
`temperature` `number`
Регулирует степень случайности результатов работы модели. Чем выше значение, тем более креативными будут ответы, а чем ниже — тем более детерминированными.
​
`max_tokens` `number`
Ограничивает общее количество токенов в ответе, фактически регулируя длину вывода.
​
`timeout` `number`
Максимальное время (в секундах), в течение которого можно ожидать ответа от модели, прежде чем отменить запрос.
​

`max_retries` `numberdefault:"6"`
Максимальное количество попыток, которые система предпримет для повторной отправки запроса в случае сбоя из-за таких проблем, как тайм-ауты в сети или ограничения скорости. При повторных попытках используется экспоненциальная задержка с джиттером. Сетевые ошибки, ограничения скорости (429) и ошибки сервера (5xx) обрабатываются автоматически. Клиентские ошибки, такие как 401 (неавторизованный доступ) или 404, не обрабатываются повторно. Для длительных агентских задач в ненадежных сетях рекомендуется увеличить это значение до 10–15.
С помощью `init_chat_model` передайте эти параметры как встроенные **kwargs.

Каждая интеграция с чат-моделью может иметь дополнительные параметры, используемые для управления специфическими для провайдера функциями.
Например, `ChatOpenAI` имеет `use_responses_api` возможность указывать, следует ли использовать API ответов или дополнений OpenAI.
Чтобы узнать обо всех параметрах, поддерживаемых конкретной чат-моделью, перейдите на страницу интеграции с чат-моделями.

## Вызов
Для получения результата необходимо вызвать чат-модель. Существует три основных способа вызова, каждый из которых подходит для разных сценариев использования.

### Invoke
Самый простой способ вызвать модель — использовать `invoke()` с одним сообщением или списком сообщений.

Список сообщений можно передать модели чата для отображения истории переписки. У каждого сообщения есть роль, которую модели используют, чтобы указать, кто отправил сообщение в переписке.
Подробнее о ролях, типах и содержании см. в руководстве по сообщениям.

Если тип возвращаемого значения вашего запроса — строка, убедитесь, что вы используете чат-модель, а не языковую модель. Устаревшие языковые модели для автозаполнения текста возвращают строки напрямую. Чат-модели `LangChain` имеют префикс `«Chat»`, например `ChatOpenAI(/oss/integrations/chat/openai)`.

### Stream
Большинство моделей могут передавать выходные данные по мере их генерации. Благодаря постепенному отображению выходных данных потоковая передача значительно улучшает пользовательский опыт, особенно при получении длинных ответов.
При вызове `stream()` возвращается итератор, который выдает выходные данные по мере их генерации. Для обработки каждого фрагмента в режиме реального времени можно использовать цикл.
В отличие от `invoke()`, которая возвращает один `AIMessage` после того, как модель сгенерирует полный ответ, `stream()` возвращает несколько `AIMessageChunk` объектов, каждый из которых содержит часть выходного текста. Важно отметить, что каждый фрагмент потока предназначен для объединения в полное сообщение путем суммирования.
Полученное сообщение можно обрабатывать так же, как сообщение, сгенерированное с помощью `invoke()` — например, его можно добавить в историю сообщений и передать обратно в модель в качестве контекста диалога.
Потоковая передача возможна только в том случае, если все этапы программы умеют обрабатывать поток фрагментов. Например, приложение, не поддерживающее потоковую передачу, должно сохранять весь результат в памяти, прежде чем его можно будет обработать.

### Batch
Пакетная обработка набора независимых запросов к модели может значительно повысить производительность и снизить затраты, поскольку обработка может выполняться параллельно.
В этом разделе описан метод `batch()` для чат-модели, который распараллеливает вызовы модели на стороне клиента.
Он отличается от пакетных API, поддерживаемых поставщиками логического вывода, такими как OpenAI или Anthropic.
По умолчанию `batch()` возвращает только итоговый результат для всего пакета. Если вы хотите получать результат для каждого отдельного входного параметра по мере его обработки, вы можете использовать потоковую передачу результатов с помощью `batch_as_completed()`.
При использовании batch_as_completed() результаты могут приходить не по порядку. Каждый результат содержит входной индекс для сопоставления, чтобы при необходимости восстановить исходный порядок.
При обработке большого количества входных данных с помощью `batch()` или `batch_as_completed()` может потребоваться ограничение максимального количества параллельных вызовов. Это можно сделать, задав атрибут `max_concurrency` в словаре `RunnableConfig`.
Полный список поддерживаемых атрибутов приведен в справочнике `RunnableConfig`

## Вызов инструмента
Модели могут запрашивать вызов инструментов, которые выполняют такие задачи, как получение данных из базы данных, поиск в интернете или запуск кода. Инструменты представляют собой сочетание:
1. Схема, включающая название инструмента, описание и/или определения аргументов (часто в виде схемы JSON)
2. Функция или корутина для выполнения.
Возможно, вы слышали термин «вызов функции». Мы используем его как синоним термина «вызов инструмента».
Чтобы сделать определенные вами инструменты доступными для использования в модели, необходимо связать их с помощью `bind_tools`. При последующих вызовах модель может при необходимости вызывать любой из связанных инструментов.
Некоторые поставщики моделей предлагают встроенные инструменты, которые можно включить с помощью параметров модели или вызова (например, ChatOpenAI, ChatAnthropic). Подробности см. в соответствующей справочной информации о поставщике.
При привязке пользовательских инструментов ответ модели включает в себя запрос на выполнение инструмента. При использовании модели отдельно от агента вы сами должны выполнить запрошенный инструмент и вернуть результат в модель для использования в последующих рассуждениях. При использовании агента цикл агента сам выполнит цикл выполнения инструмента.

Цикл выполнения инструмента
Когда модель возвращает вызовы инструментов, вам нужно выполнить эти инструменты и передать результаты обратно в модель. Таким образом создается цикл взаимодействия, в котором модель может использовать результаты работы инструментов для формирования окончательного ответа. LangChain включает в себя агенты, которые выполняют эту задачу за вас.
Каждый ToolMessage результат, возвращаемый инструментом, содержит tool_call_id , соответствующий исходному вызову инструмента, что помогает модели сопоставлять результаты с запросами.

Принудительные вызовы инструментов
По умолчанию модель может самостоятельно выбирать инструмент в зависимости от действий пользователя. Однако вы можете принудительно выбрать инструмент, чтобы модель использовала либо конкретный инструмент, либо любой инструмент из заданного списка.

Параллельные вызовы инструментов
Многие модели при необходимости поддерживают параллельный вызов нескольких инструментов. Это позволяет модели одновременно собирать информацию из разных источников.
Модель интеллектуально определяет, когда целесообразно использовать параллельное выполнение, исходя из независимости запрашиваемых операций.
В большинстве моделей, поддерживающих вызов инструментов, по умолчанию включена функция параллельного вызова инструментов. В некоторых моделях (в том числе OpenAI и Anthropic) эту функцию можно отключить. Для этого установите parallel_tool_calls=False:

Вызовы инструментов потоковой передачи
При потоковой передаче ответов вызовы инструментов формируются постепенно с помощью ToolCallChunk. Это позволяет видеть вызовы инструментов по мере их выполнения, а не ждать полного ответа.
Вы можете накапливать фрагменты, чтобы создавать полноценные вызовы инструментов.

## Структурированный вывод
Можно настроить модели так, чтобы они выдавали ответ в формате, соответствующем заданной схеме. Это удобно для последующего анализа и использования вывода. LangChain поддерживает несколько типов схем и методов для обеспечения структурированного вывода:
- Pydantic
Модели Pydantic предоставляют самый широкий набор функций, включая проверку полей, описания и вложенные структуры.
- TypedDict
TypedDict в Python — это более простая альтернатива моделям Pydantic, которая идеально подходит для случаев, когда не требуется проверка во время выполнения.
- JSON Schema
Предоставьте схему JSON для максимального контроля и совместимости.

Основные рекомендации по структурированному выводу
- Параметр метода: некоторые провайдеры поддерживают различные методы структурированного вывода:
 - `json_schema`: использует специальные функции структурированного вывода, предлагаемые провайдером.
 - `function_calling`: формирует структурированный вывод, вызывая инструмент в соответствии с заданной схемой.
 - `json_mode`: предшественник `json_schema`, предлагаемый некоторыми провайдерами. Генерирует корректный JSON, но схема должна быть описана в подсказке.
- Включить необработанный текст: установите include_raw=True, чтобы получить как обработанный результат, так и необработанное сообщение ИИ.
- Валидация: модели Pydantic обеспечивают автоматическую валидацию.  TypedDict и JSON Schema требуют ручной валидации.
Поддерживаемые методы и параметры конфигурации см. на странице интеграции вашего провайдера.

## Дополнительные сведения

### Профили моделей
**Для профилей моделей требуется langchain>=1.1.**
Чат-модели LangChain могут предоставлять словарь поддерживаемых функций и возможностей через profile атрибут.
Большая часть данных о профиле модели получена в рамках проекта models.dev — инициативы с открытым исходным кодом, предоставляющей данные о возможностях моделей. Эти данные дополнены новыми полями для использования в LangChain. Эти дополнения соответствуют требованиям основного проекта по мере его развития.
Данные о профиле модели позволяют приложениям динамически работать с возможностями моделей. Например:
1. Промежуточное ПО для обобщения может запускать обобщение в зависимости от размера контекстного окна модели.
2. Структурированный вывод в `create_agent` может определяться автоматически (например, путем проверки поддержки встроенных функций структурированного вывода).
3. Входные данные модели могут быть ограничены в зависимости от поддерживаемых модальностей и максимального количества входных токенов.
4. Deep Agents CLI фильтрует интерактивный переключатель моделей в пользу моделей, в профилях которых указана поддержка `tool_calling` и текстового ввода-вывода, а также отображает размеры контекстного окна и флаги возможностей в подробном представлении селектора.

    Обновление или перезапись данных профиля

    Данные профиля модели можно изменить, если они отсутствуют, устарели или неверны.

    Вариант 1 (быстрое решение)
    Вы можете создать экземпляр модели чата с любым допустимым профилем.
    `profile` также является обычным `dict` и может быть обновлен на месте. Если экземпляр модели используется совместно, рассмотрите возможность использования `model_copy`, чтобы избежать изменения общего состояния.

    Вариант 2 (исправление данных в вышестоящем источнике)
    Основным источником данных является проект models.dev. Эти данные объединяются с дополнительными полями и переопределениями в интеграционных пакетах LangChain и поставляются вместе с ними.
    Данные профиля модели можно обновить следующим образом:
    1. (При необходимости) обновите исходные данные в models.dev с помощью запроса на включение изменений в репозиторий на GitHub.
    2. (При необходимости) обновите дополнительные поля и переопределения в `langchain_<package>/data/profile_augmentations.toml` с помощью запроса на включение изменений в интеграционный пакет LangChain`.
    3. Используйте инструмент `langchain-model-profiles CLI`, чтобы получить последние данные с models.dev, добавить улучшения и обновить данные профиля.
**Профили моделей — это бета-функция. Формат профиля может быть изменен.**

### Мультимодальность
Некоторые модели могут обрабатывать и возвращать нетекстовые данные, такие как изображения, аудио и видео. Вы можете передать модели нетекстовые данные, предоставив блоки контента.
Все чат-модели LangChain с мультимодальными возможностями поддерживают:
1. Данные в стандартном межплатформенном формате (см. наше руководство по работе с сообщениями)
2. Формат OpenAI дополнения в чате
3. Любой формат, характерный для конкретной платформы (например, модели Anthropic принимают собственный формат Anthropic)
Подробнее см. в разделе о мультимодальных данных руководства по работе с сообщениями.
Некоторые модели могут возвращать мультимодальные данные в составе ответа. Если это необходимо, в результате `AIMessage` будут получены блоки контента с мультимодальными типами.

### Рассуждения
Многие модели способны выполнять многоэтапные рассуждения для получения конечного результата. Это предполагает разбиение сложных задач на более мелкие и выполнимые этапы.

Если это поддерживается базовой моделью, вы можете просмотреть процесс рассуждений, чтобы лучше понять, как модель пришла к окончательному ответу.

В зависимости от модели вы можете указать, сколько усилий она должна приложить для логического вывода. Аналогичным образом вы можете попросить модель полностью отключить логический вывод. Это может быть реализовано в виде категориальных «уровней» логического вывода (например, `low` или `high`) или целочисленных бюджетов токенов.
Подробнее см. на странице интеграции или в справочнике для вашей модели чата.

### Локальные модели
LangChain поддерживает локальное выполнение моделей на вашем собственном оборудовании. Это полезно в тех случаях, когда конфиденциальность данных имеет решающее значение, когда вы хотите использовать собственную модель или избежать затрат, связанных с использованием облачной модели.

Ollama — один из самых простых способов локального запуска чат-ботов и встраиваемых моделей.

### Кэширование подсказок
Многие провайдеры предлагают функции кэширования подсказок, чтобы сократить время ожидания и затраты на повторную обработку одних и тех же токенов. Эти функции могут быть неявными или явными:
- Неявное кэширование подсказок: провайдеры автоматически передают информацию об экономии средств, если запрос попадает в кэш.
- Явное кэширование: провайдеры позволяют вручную указывать точки кэширования для большего контроля или гарантированной экономии средств.

Кэширование подсказок часто включается только при достижении минимального порога входных токенов. Подробнее см. на страницах провайдеров.

Использование кэша будет отражено в метаданных об использовании ответа модели.

### Использование инструментов на стороне сервера
Некоторые провайдеры поддерживают циклы вызова инструментов на стороне сервера: модели могут взаимодействовать с веб-поиском, интерпретаторами кода и другими инструментами и анализировать результаты за один обмен данными.
Если модель вызывает инструмент на стороне сервера, в ответном сообщении будет содержаться информация о вызове и результатах работы инструмента. При обращении к блокам контента в ответе будут возвращены вызовы инструментов на стороне сервера и результаты их работы в формате, не зависящем от провайдера.

Это один из этапов диалога; нет связанных объектов ToolMessage, которые необходимо передавать, как при вызове инструмента на стороне клиента.

Ознакомьтесь с страницей интеграции вашего провайдера, чтобы узнать о доступных инструментах и особенностях их использования.

### Ограничение скорости
Многие поставщики чат-моделей устанавливают ограничение на количество вызовов, которые можно сделать за определенный период времени. Если вы превысите лимит, то, как правило, получите от поставщика ответ об ошибке, связанной с ограничением скорости, и вам придется подождать, прежде чем отправлять новые запросы.

Для управления ограничениями скорости интеграции с чат-моделями поддерживают параметр `rate_limiter`, который можно указать при инициализации, чтобы контролировать частоту запросов.

LangChain поставляется со встроенным (необязательным) ограничителем `InMemoryRateLimiter`. Этот ограничитель является потокобезопасным и может использоваться несколькими потоками в рамках одного процесса.
Предоставленный ограничитель скорости может ограничивать только количество запросов в единицу времени. Он не поможет, если вам нужно ограничить количество запросов в зависимости от их размера.

### Базовый URL и настройки прокси
Вы можете настроить собственный базовый URL для провайдеров, использующих API OpenAI Chat Completions.

`model_provider="openai"` (или прямое ChatOpenAI использование) ориентировано на официальную спецификацию API OpenAI. Поля, специфичные для провайдеров, в маршрутизаторах и прокси-серверах могут не извлекаться и не сохраняться.

- Пользовательский базовый URL-АДРЕС
Многие поставщики моделей предлагают API, совместимые с OpenAI (например, Together AI, vLLM). Вы можете использовать init_chat_model с этими поставщиками, указав соответствующий параметр base_url.
При использовании экземпляра класса модели прямого чата имя параметра может отличаться в зависимости от провайдера.

- Конфигурация HTTP-прокси
Для развертываний, требующих использования HTTP-прокси, некоторые интеграции с моделями поддерживают настройку прокси.
Поддержка прокси зависит от интеграции. Ознакомьтесь с справочной информацией конкретного поставщика моделей, чтобы узнать о возможностях настройки прокси.

### Логарифмические вероятности
Некоторые модели можно настроить на возврат логарифмических вероятностей на уровне токенов, представляющих вероятность появления того или иного токена, задав параметр logprobs при инициализации модели.

### Использование токенов
Некоторые поставщики моделей возвращают информацию об использовании токенов в ответе на запрос. Если такая информация доступна, она будет включена в `AIMessage` объекты, созданные с помощью соответствующей модели. Подробнее см. в руководстве по сообщениям.
Некоторые API-интерфейсы провайдеров, в частности чат-боты OpenAI и Azure OpenAI, требуют от пользователей согласия на получение данных об использовании токенов в потоковом режиме. Подробнее см. в разделе метаданные об использовании в потоковом режиме руководства по интеграции.

Вы можете отслеживать общее количество токенов в моделях приложения с помощью обратного вызова (from langchain_core.callbacks import UsageMetadataCallbackHandler) или контекстного менеджера (from langchain_core.callbacks import get_usage_metadata_callback).

### Конфигурация вызова
При вызове модели вы можете передать дополнительную конфигурацию через параметр `config` с помощью словаря `RunnableConfig`. Это позволяет во время выполнения управлять поведением модели, обратными вызовами и отслеживанием метаданных.

Эти значения конфигурации особенно полезны в следующих случаях:
- Отладка с помощью LangSmith трассировки
- Реализация пользовательского ведения журнала или мониторинга
- Контроль использования ресурсов в рабочей среде
- Отслеживание вызовов в сложных конвейерах

Ключевые атрибуты конфигурации^
- `run_name` string
Определяет этот конкретный вызов в журналах и трассировках. Не наследуется подвызовами.
- `tags` string[]
Метки, наследуемые всеми подвызовами для фильтрации и организации в инструментах отладки.
- `metadata` object
Пользовательские пары «ключ-значение» для отслеживания дополнительного контекста, наследуемого всеми подвызовами.
- `max_concurrency` number
Определяет максимальное количество параллельных вызовов при использовании `batch()` или `batch_as_completed()`.
- `callbacks` array
Обработчики для отслеживания событий во время выполнения и реагирования на них.
- `recursion_limit` number
Максимальная глубина рекурсии для цепочек, предотвращающая возникновение бесконечных циклов в сложных конвейерах.

См. полный RunnableConfig список всех поддерживаемых атрибутов.

### Настраиваемые модели

Вы также можете создать настраиваемую во время выполнения модель, указав `configurable_fields`. Если вы не укажете значение модели, то `model` и `model_provider` будут настраиваемыми по умолчанию.

- Настраиваемая модель со значениями по умолчанию
Мы можем создать настраиваемую модель со значениями по умолчанию, указать, какие параметры являются настраиваемыми, и добавить префиксы к настраиваемым параметрам.

- Декларативное использование настраиваемой модели
Мы можем вызывать декларативные операции, такие как `bind_tools`, `with_structured_output`, `with_configurable` и т. д., для настраиваемой модели и объединять настраиваемые модели в цепочку так же, как мы объединяли бы объекты модели чата, созданные с помощью регулярного инстанцирования.
